# 测试分层

## 金字塔模型

```
┌─────────────────────────────────────────┐
│ E2E (10%)        完整业务闭环           │  用户下单 → 审批 → 支付 → 发货
├─────────────────────────────────────────┤
│ Integration (30%) 模块间真实依赖链        │  A → B → C → D 完整流程
├─────────────────────────────────────────┤
│ API / UI (60%)    单接口/单页面逻辑      │  单 endpoint / 单页面的正向+负向
└─────────────────────────────────────────┘
```

---

## API 层 — 单接口测试（60%）

**目标**：验证单个接口自身逻辑正确，不依赖其他模块的真实数据。

**覆盖内容**：

| 类型 | 说明 | 示例 |
|------|------|------|
| 正向 | 正常参数，预期成功 | 创建用户时传完整参数 |
| 边界 | 临界值、空值、超长 | 用户名为空、超长字符串 |
| 异常 | 缺少必填、无效格式、无权限 | 缺少 email 字段、未登录 |

**BDD 写法**：

```gherkin
功能: 用户管理

  背景:
    假如 管理员已登录

  @smoke @positive
  场景: 获取所有用户列表
    当 GET "/users"
    那么 响应状态码应该为 200

  @positive
  场景: 创建新用户
    假如 存在"用户"
    那么 数据库中存在"用户"

  @negative
  场景: 创建用户时缺少必填字段
    当 POST "/users":
    """
    { "email": "test@example.com" }
    """
    那么 响应状态码应该为 422
```

**pytest 写法**：

```python
class TestUser:
    @allure.title("获取所有用户列表")
    def test_get_all_users(self, api_client):
        resp = api_client.get("/users")
        assert resp["code"] == 200

    @allure.title("创建新用户")
    def test_create_user(self, user_builder):
        user = user_builder.create()
        assert user.id is not None

    @allure.title("创建用户时缺少必填字段")
    def test_create_user_missing_field(self, api_client):
        resp = api_client.post("/users", {"email": "test@example.com"})
        assert resp["code"] == 422
```

## Integration 层 — 模块集成测试（30%）

**目标**：验证多个模块串联的真实依赖链路。

**覆盖内容**：

| 类型 | 说明 | 示例 |
|------|------|------|
| 跨模块流程 | A 创建 → B 依赖 A 的数据 → C 依赖 B 的结果 | 报销审批链 |
| 数据传递 | 上一个接口的返回值作为下一个的输入 | `${user.id}` 占位符 |
| 状态流转 | 实体状态随流程变化 | pending → approved → paid |

**BDD 写法**：

```gherkin
功能: 报销审批流程

  背景:
    假如 管理员已登录

  @integration @positive
  场景: 完整审批流程
    # Step 1: 创建用户
    假如 存在"用户"
    # Step 2: 提交报销（依赖用户ID）
    当 使用用户ID POST "/reimbursements":
    """
    {
      "user_id": ${user.id},
      "amount": 5000,
      "reason": "差旅费"
    }
    """
    # Step 3: 部门审批 → 财务审批
    当 POST "/reimbursements/${reimbursement.id}/dept-approve":
    """
    { "status": "approved" }
    """
    当 POST "/reimbursements/${reimbursement.id}/finance-approve":
    """
    { "status": "approved" }
    """
    那么 响应状态码应该为 200
    而且 报销状态应该为 "paid"
```

**pytest 写法**：

```python
class TestApprovalFlow:
    def test_full_approval(self, full_approval_scenario):
        result = full_approval_scenario.run()
        assert result.status == "paid"
```

## E2E 层 — 端到端测试（10%）

**目标**：验证完整业务闭环，从用户操作到最终结果。

**特点**：场景长、依赖全、跑得慢。**只在关键业务路径上写**。

**BDD 写法**：

```gherkin
功能: 订单全流程

  @e2e
  场景: 用户下单到收货全流程
    假如 存在"用户"
    当 GET "/products"
    那么 响应数据应该是列表
    当 使用用户ID POST "/orders":
    """
    { "product_id": ${product.id}, "quantity": 2 }
    """
    当 POST "/orders/${order.id}/pay"
    当 POST "/orders/${order.id}/ship"
    当 POST "/orders/${order.id}/confirm"
    那么 订单状态应该为 "completed"
```

## Marker 与 CI 策略

| 标签 | 层级 | 占比 | 运行策略 |
|------|------|------|---------|
| `@smoke` | API 正向 | 60% | 每次提交 / PR 必跑 |
| `@positive` | API 正向 | — | 每次提交 / PR 必跑 |
| `@negative` | API 负向 | — | 每次提交 / PR 必跑 |
| `@integration` | Integration | 30% | 合并到主分支时跑 |
| `@e2e` | E2E | 10% | 定时跑 / 发布前跑 |

```bash
pytest -m smoke                  # 冒烟（每次提交）
pytest -m "not e2e"              # 日常跑（跳过 E2E）
pytest -m "integration or e2e"   # 全量跑（发版前）
```

## 设计原则

| 原则 | 说明 |
|------|------|
| 一个场景只测一件事 | 不要在一个用例里既测创建又测查询 |
| 正向至少 1 条 | 每个接口/页面至少一个正向场景 |
| 负向至少 1 条 | 覆盖参数校验、权限校验等异常 |
| 不硬编码 | 用 `${变量}` 或数据工厂生成唯一值 |

### 场景命名

```
查询类：获取所有XX列表、根据ID获取指定XX
创建类：创建新XX、批量创建XX
更新类：更新XX信息、更新XX状态
删除类：删除XX、批量删除XX
异常类：创建XX时缺少必填字段、使用无效ID获取XX
```

### 断言规范

每个场景至少断言两条：

```
响应状态码应该为 200    # HTTP 状态码
code 为 200            # 业务返回码
```

查询额外断言数据结构（列表非空、字段存在），写入额外断言落地（数据库中存在）。
